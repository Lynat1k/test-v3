## 0. Жёсткие правила

- **Только HTML5 Canvas 2D Context.** НЕ TradingView Lightweight Charts. НЕ WebGL на старте.
- Цель производительности: **стабильные 60 FPS** при потоке realtime-апдейтов.
- Никаких фейковых данных. Источник: WS-хаб бэкенда (realtime) + REST (история, догрузка).
- Тепловую карту (heatmap) НЕ реализуем сейчас. Только структуру под неё готовит БД.
- По спорным UI-решениям движка — спрашивать пользователя ДО реализации.

## 2. Структуры данных (контракт с бэкендом)

Бэкенд отдаёт JSON в этом формате (совпадает с типами PROCLUSTER3):

```typescript
interface ClusterCell {
  price: number;   // нижняя граница бина (binPriceLow)
  bid: number;     // рыночные продажи (округл. 0.1)
  ask: number;     // рыночные покупки (округл. 0.1)
  volume: number;  // bid + ask
  isPoc: boolean;
  isBuyImbalance: boolean;   // ask[price] > bid[price-1] * ratio (диагональ, см. §6)
  isSellImbalance: boolean;  // bid[price] > ask[price+1] * ratio
}

interface ClusterCandle {
  time: number;          // открытие свечи (unix ms, UTC, trade time)
  open: number;
  high: number;
  low: number;
  close: number;
  volume: number;        // Σ cell.volume
  delta: number;         // Σ (ask − bid)
  cells: ClusterCell[];  // отсортированы по price
}
```

> Движок НЕ считает кластеры сам — он рисует готовое. POC/VAH/VAL он может
> досчитать на клиенте для отображения (как сейчас), но bid/ask/volume — только с бэка.

## 3. Отвязка от фейковых данных + подключение к бэкенду

Удалить/заменить:
- Все вызовы `dataGenerator` внутри движка и родительских компонентов.
- `src/lib/api.ts` (`getClusterCandles`) — перенаправить на реальный REST бэкенда.

Источники данных движка:
- **Первичная загрузка**: REST `GET /api/candles?symbol=&market=&tf=&compression=` → последние ~700 свечей (из Redis-кэша бэкенда).
- **Догрузка истории при прокрутке влево**: REST `GET /api/candles?...&before=<time>&limit=N`.
- **Realtime**: WebSocket — апдейты текущей (и закрытых) свечей (см. §8).

Правило кэша: движок держит в памяти видимый диапазон + буфер. При прокрутке за пределы буфера — запрос догрузки. НЕ грузить весь месяц/год сразу.

## 4. Производительность — обязательные требования (для 60 FPS)

Это главная причина, по которой реальные данные могут «уронить» FPS. Реализовать строго:

1. **Рендер только через `requestAnimationFrame`.** Один общий rAF-цикл. Никаких перерисовок напрямую из обработчиков WS/событий.
2. **Dirty-флаг.** Перерисовывать кадр ТОЛЬКО если `isDirty === true` (изменились данные, зум, скролл, ресайз). Если ничего не менялось — кадр пропускается, CPU отдыхает.
3. **Батчинг WS-апдейтов.** Входящие сообщения WS накапливать в буфер и применять РАЗ В КАДР (в начале rAF), затем ставить dirty. Не перерисовывать на каждое сообщение.
4. **Отсечение невидимого.** Рисовать только свечи в диапазоне [startIdx, endIdx] видимой области + расчёт min/max цены по видимому участку (как уже сделано в PROCLUSTER3).
5. **DPR / Retina.** Canvas масштабировать под `devicePixelRatio`, но кэшировать значение, не пересчитывать каждый кадр.
6. **Текст bid/ask.** Рисовать числа в ячейках только когда `candleWidth` и высота ячейки достаточны (порог по пикселям). Иначе пропускать — это самая дорогая операция.
7. **Оффскрин-слои (опционально, если не хватит FPS).** Статику (сетка, оси) можно держать на отдельном offscreen-canvas и перерисовывать реже, чем динамику.
8. **Никаких аллокаций в горячем цикле.** Переиспользовать объекты/массивы, не создавать новые в rAF.

> Acceptance: при активном realtime-потоке и видимых 200+ свечах в footprint-режиме — стабильные 60 FPS на целевом железе. Замерять через FPS-счётчик (dev-режим).

## 5. Управление графиком мышью (добавить к существующему)

- **Свободное перемещение** графика (drag) — есть, сохранить.
- **Колесо мыши** → зум in/out **к точке под курсором** (anchor на позиции мыши, чтобы цена/время под курсором оставались на месте).
- **SHIFT + колесо** → растяжение по **вертикали** (ось цены Y) от точки курсора.
- **CTRL + колесо** → растяжение по **горизонтали** (ось времени X) от точки курсора.
- Отдельный zoom по оси Y (цена) и по оси X (время) — drag по самим осям.

Формула зума к курсору (псевдокод):
```
worldX_before = screenToWorldX(mouseX)
scaleX *= zoomFactor            // >1 при zoom in
worldX_after = screenToWorldX(mouseX)
offsetX += (worldX_after - worldX_before)   // компенсация, чтобы точка не "уехала"
setDirty()
```
Аналогично для Y с `scaleY` / `offsetY`.

## 6. Диагональный имбаланс (стиль ATAS)

В существующем движке имбаланс горизонтальный (Bid vs Ask на одном уровне). По ТЗ нужен **диагональный**, как в ATAS:

```
Bid Imbalance (зелёный):  ask[price]  >  bid[price - 1] * (ratio)
Sell Imbalance (красный): bid[price]  >  ask[price + 1] * (ratio)
```
где `ratio` соответствует порогу **> 300%** (т.е. одна сторона больше другой в 3 раза и более).

Подсветка: цифры ask красить зелёным при bid-имбалансе, цифры bid красить красным при sell-имбалансе (диагональное сравнение соседних ценовых уровней).

> price−1 / price+1 — это соседний бин по цене (шаг = binSize текущего сжатия), а не соседний тик.

## 7. Режим "Авто" переключения вида свечей

Кнопка "Auto" в панели типов свечей. Переключение по количеству свечей в видимой области:

| Видимых свечей | Вид |
|---|---|
| < 70 | **Кластера** (footprint с bid/ask по уровням) |
| 70 – 300 | **Футпринт + гистограмма объёма** |
| ≥ 300 | **Японские свечи** |

> Пороги вынести в константы (легко менять). По умолчанию: 70 и 300.
> Ручные режимы (Японские / Бары / Footprint / Clusters) переопределяют Auto.

## 8. Realtime через WebSocket-хаб бэкенда

- Бэкенд шлёт обновлённый кластер текущей свечи каждые **100–500 ms** (батч на стороне бэка).
- Формат сообщения: дельта-апдейт текущей свечи (или полная свеча при закрытии).
- Движок: применить апдейт в буфер → раз в кадр обновить данные → setDirty.
- При закрытии свечи бэкенд шлёт финальную свечу + открывает новую.
- Реконнект WS: после переподключения запросить REST «свечи после <last_time>», чтобы добрать пропуск (бэкенд гарантирует целостность по tradeId на своей стороне).

## 9. Уровни сжатия в движке

- Переключатель сжатия в верхней панели (Header). Значения берутся из конфига тикера: futures 25/50/75…250, spot 500/1000/1500…5000 (до 10 уровней).
- При смене сжатия движок запрашивает у бэкенда данные нужного уровня (бэкенд отдаёт из ClickHouse базовый уровень или объединённый высший — см. docs/01 §3, §7).
- По умолчанию для каждого ТФ — уровень сжатия из админки.

## 10. Workspace (рабочие пространства)

Кнопка выбора в углу окна графика:
- 1 график
- 2 графика по горизонтали
- 2 графика по вертикали

Требования:
- Перетаскиваемый **разделитель** между двумя графиками меняет их соотношение размеров.
- Каждый график — независимый инстанс движка (свой тикер/ТФ/сжатие/вид).
- Окно чарта автоматически заполняет всю свободную область экрана.
- Если стакан (DOM) свёрнут в край — график растягивается до конца экрана (ресайз → setDirty + пересчёт canvas).
