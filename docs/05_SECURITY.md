# docs/05_SECURITY.md — Безопасность PROCLUSTER

> Читать перед фазой 14 (аудит) и точечно после фаз 2,6,7,9,11.
> Аудит безопасности — ОБЯЗАТЕЛЬНАЯ отдельная фаза в режиме plan перед продакшеном.
> Принцип: фронту НЕ доверяем, решение всегда за бэком.

---

## 1. SSH / сервер (хардненинг VPS)

- Вход по SSH-ключам, парольный вход ОТКЛЮЧИТЬ (`PasswordAuthentication no`).
- Root-логин запретить (`PermitRootLogin no`), работать под отдельным sudo-юзером.
- Сменить порт SSH с 22 на нестандартный (снижает шум ботов).
- **fail2ban**: бан IP после N неудачных попыток SSH (анти-брутфорс).
- UFW/iptables: открыть только 80, 443, (нестандартный SSH). Всё остальное закрыть.
- ClickHouse / Redis / SQLite — НЕ выставлять наружу. Только внутри docker-сети.
- Автообновления безопасности ОС (unattended-upgrades).

---

## 2. Сеть / reverse-proxy (Caddy)

- Caddy терминирует TLS, авто-HTTPS для procluster.online, chart.procluster.online.
- Принудительный HTTPS-редирект, HSTS.
- **Rate-limit на уровне Caddy** (на старте основной заслон):
  - на /api/auth/* — жёсткий лимит (анти-брутфорс логина)
  - на /api/* и WS — общий лимит запросов с одного IP (анти-DDoS L7)
- Заголовки безопасности: CSP, X-Frame-Options, X-Content-Type-Options,
  Referrer-Policy, Permissions-Policy.
- CORS: разрешить ТОЛЬКО procluster.online и chart.procluster.online (не `*`).

---

## 3. Защита от перегрузки (анти-DDoS / анти-flood)

- Rate-limit на бэке (Go) поверх Caddy: per-IP и per-user токен-бакет.
- Лимит на WS: макс. число соединений с IP, макс. подписок на соединение.
- Таймауты на все хендлеры (read/write/idle), ограничение размера тела запроса.
- Backpressure на WS-хабе: медленный клиент не тормозит остальных (буфер + дроп/дисконнект).
- Защита тяжёлых эндпоинтов (загрузка истории, большие диапазоны свечей):
  лимит диапазона, очередь, только Admin для загрузки истории.
- Graceful degradation: при перегрузке отдавать 429, не падать.

---

## 4. Аутентификация / сессии

- Пароли: хеш **bcrypt/argon2** (НИКОГДА не хранить в открытом виде).
- JWT: короткий access + refresh; секрет в env, не в коде. Ротация секрета возможна.
- Google OAuth: проверять `state` (анти-CSRF), валидировать id_token.
- Брутфорс логина: лимит попыток + экспоненциальная задержка + лок аккаунта/IP.
- Смена пароля: требовать текущий пароль; инвалидировать старые сессии.
- Сессии: httpOnly + Secure + SameSite cookie (если cookie); защита от XSS-кражи токена.

---

## 5. Авторизация (роли)

- ВСЕ admin-эндпоинты: проверка `role=admin` на бэке (middleware), не только скрытие в UI.
- Лимиты тарифов (история, сжатие, Cluster Search, TG) — решает бэк (см. docs/04).
- IDOR-защита: пользователь не может читать/менять чужой профиль/настройки по id.
- Никогда не доверять роли/лимиту/user_id, присланным с клиента.

---

## 6. Валидация ввода / инъекции

- Все входные параметры API строго валидировать (symbol, market, tf, compression,
  диапазоны, числа фильтров Cluster Search) — whitelisting, не blacklisting.
- **ClickHouse**: только параметризованные запросы. НЕ конкатенировать строки в SQL.
  symbol/tf/market сверять с whitelist (ticker_config), а не подставлять напрямую.
- **SQLite**: только prepared statements.
- Telegram-токены пользователей: валидировать формат, хранить шифрованно (at rest).
- Защита от XSS на фронте: не вставлять пользовательский ввод как HTML;
  React экранирует по умолчанию — не использовать dangerouslySetInnerHTML.

---

## 7. Секреты и конфиг

- Все секреты (JWT, OAuth client secret, Binance ключи, БД-пароли) — в env / docker secrets.
- `.env` в `.gitignore`. В репо только `.env.example` без значений.
- Никаких секретов и токенов в логах. Маскировать PII/токены в логах.
- Проверить, что старые ключи из PROCLUSTER3 (если были) удалены/отозваны.

---

## 8. Данные / логи

- ClickHouse логи жёстко ограничены TTL + ротация (см. docs/01 §8) — это и безопасность диска.
- Логи приложения: ротация по размеру, ограниченное хранение.
- Бэкапы SQLite (users) — регулярно, шифрованно, доступ ограничен.
- GDPR-минимум: удаление аккаунта чистит персональные данные.

---

## 9. Зависимости / supply chain

- Аудит зависимостей: `npm audit` (фронт), `govulncheck` (Go).
- Убрать неиспользуемые пакеты (в т.ч. `@google/genai` из PROCLUSTER3).
- Пинить версии в lock-файлах. Регулярно обновлять критичные патчи.
- Docker-образы из доверенных источников, обновлять базовые образы.

---

## 10. CI/CD безопасность

- Секреты CI — в защищённом хранилище, не в YAML.
- Шаги в пайплайне: lint → тесты → `npm audit`/`govulncheck` → build → deploy.
- Деплой по защищённому каналу (ключи деплоя ограниченных прав).
- Не деплоить при провале security-шагов.

---

## 11. Процесс аудита (фаза 14)

- Режим **plan** (read-only), субагенты `@security-auditor` + `@code-reviewer`.
- Отчёт в `docs/SECURITY_AUDIT.md`: находки по severity (critical/high/medium/low) + фикс.
- Фиксы — отдельной build-сессией после ревью.
- Повторный прогон после фиксов.
